Semantic HTML

Show / Hide

Semantics refers to the meaning of something. When using HTML one should think about the information within the template and the organization of that information rather than its appearance. Once the data is organized it can later be styled using CSS for presentation purposes. The semantics of a web page are important to aid in programmatically traversing the data. Web browsers use this data and so do other agents such as smart phones and web crawlers for instance. HTML is the data structure for the web page and is separate from the graphical aspects we interact with as users.

Along with the idea of the separation between data and presentation is that HTML elements also need to be used to represent the data appropriately. This is to help ensure that web browsers and other agents (like screen readers) navigate and access the HTML correctly. For example if an agent wanted to display the title of the page it would look within the HEAD element for a child TITLE tag. This consistency is something we need to be able to rely upon and is why it is also so important to remember.

The following sections describe how semantic HTML plays an important role in Search Engine Optimization, Screen Readers, and Layout Containers.

Search Engine Optimization

Show / Hide

Optimizing a web page to increase its search index on search engines, such as Google and Bing, is important in grabbing a consumer's eye. Not often is it that someone reviews several pages of search result to find what they're looking for. Algorithms are very good at locating what it is exactly that that person is looking for and it usually appears toward the top of the results. Plus there are more widgets now which provide greater functionality to the user upfront (such as when searching for airline tickets for example) users don't have to look very far to find what they need.

In order to be relevant a web page must take into consideration "SEO" techniques to increase their chances of being ranked higher and found. This includes using the correct HTML elements and attributes (such as title and alt), having relevant words and content, as well as how many other web pages link back to it which is important as a search engine's "crawler" investigates websites and links across the internet.

If a website has the same content in more than one location it will help to use the rel=canonical attribute so search engines can redirect users to the page you would rather them see. This helps search engines organize the ranks appropriately and increases the SEO. Another technique similar to this is a 301 redirect. 301 is the status code for content that has moved permanently to another URL. Other codes include 302 which is either for "moved" or "found" content. In HTTP version 1.0 it meant "moved temporarily" but in version 1.1 it's been updated to mean "found". 307 became the replacement for "moved temporarily" in version 1.1. Meta Refresh is another redirect which is generally used with countdown timers and "click here" links if the redirect doesn't occur. However it's not recommended for SEO and is slower.

Link equity is another ranking factor which refers to links passing "value and authority" from one page to another. If a very valuable site links to another site then the search engine can presume the content on the linked site is valuable as well. Links need to be relevant to the original content, crawlable (via the robots.txt file), located in prominent places (e.g. within the main content as opposed to a footer), and use the appropriate HTTP status codes (e.g. 200 and 301). These factors play into a links value and equity.

The robots.txt file is the first file a crawler will parse once it reaches a website. This file can tell the crawler which pages to skip which can be important so internal pages (those of which a user interacts with once logged into the site) are not crawled and indexed. Google has even commented that internal search results pages are considered spam so these should be excluded in this way.

Schema Markup is another aspect of SEO. It's a "semantic vocabulary of tags (or microdata) that you can add to your HTML to improve the way search engines read and represent your pages" in search results according to Moz.com. Moz.com continues by describing the differences between Schema, microdata, and structured data:

  • Structured data is a system of pairing a name with a value that helps search engines categorize and index your content. Microdata is one form of structured data that works with HTML5. Schema.org is a project that provides a particular set of agreed-upon definitions for microdata tags.

Schema can be used to enhance the search results descriptions with content such as reviews, organizational information, and information about a person or book for example. The major search engines use this data to enhance the results page but it's unclear if using the markup actually helps with search ranking but appears to increase the website's click-through rate as users are more apt to click a link with a more descriptive result.

HTML defines the semantics of a web page, and Schema helps to give context to the data. For example a header tag with the text "avatar" may refer to a movie, an image, or some other content. Using Schema helps better define the content which helps search engines determine how to display the data in search results.

The official Schema.org documentation explains that "microdata is a set of tags introduced in HTML5" which allow us to markup HTML with this additional data.

Schema Example

Some of the important tags to remember are: itemscope, itemtype, and itemprop. These can be added to HTML elements to signify different things. itemscope should be applied to the wrapping HTML element for a given group of information. The itemtype is used to describe what that group of information is, for example a movie. itemprop is used to more fully define other information about the item, for example an author's name and a genre. Below is a concrete example of apply Schema to a div describing a book. 

        <div itemscope itemtype="http://schema.org/Book">
            <h1 itemprop="name">Design Patterns: Elements of Reusable Object-Oriented Software</h1>
            <span>Written by:<span itemprop="author">Erich Gamma, Richard Helm, Ralph Johnson, John Vlissides</span></span>
            <span itemprop="genre">Software Engineering</span>
        </div>

Screen Readers

Show / Hide

Using semantic HTML is very important when considering accessibility and screen readers. The browser and accessibility tools will key off of the semantic HTML to help someone navigate your website. Using the wrong element to accomplish something, such as using a div element to create a button, can be confusing to people and other agents (like screen readers or programs parsing a website).

Using the correct ARIA and semantic HTML is also helpful in navigating a page because users can quickly navigate between links, forms, headings, paragraphs, sections and other elements. Along with that being descriptive within an opening sentence in a new paragraph for instance helps users skip content they aren't interested in much like someone could who can visually review the content.

Screen readers convert text to speech to help people navigate and understand a website. These users may have disabilities ranging from visual impairment to cognitive disabilities. Being sure to define the language the website is presented in is very important. This can be done within the html tag by defining a lang attribute. It's also possible to wrap smaller sections of text in the span element and apply the lang attribute to the span (although lang can be applied to many other elements as well).

The proper use of headings is helpful too because search engines use these to index a web page's structure and content. People are drawn to the headlines which direct them to different sections of relevant content.

Alternative text is important to have on images. This helps give context to the image when a screen reader encounters one. Screen readers will read the text in the alt attribute to the user so be sure the text is descriptive. It's also useful for users when an image link is broken because the alt text will display for them on the page. This provides some context when the path to the image is broken.

Finally it is important that links are spelled out and descriptive to the user. "Click here" means nothing but links such as "book airline tickets here" help to put things into perspective. Remember we can add title attributes to elements such as the anchor tag. Doing so is helpful because the information can be displayed as a tooltip to the user when mousing over the link.

It's important to remember that content described through a screen reader is much like an automated telephone directory. The content is "linearized" to the listener so, while some users can quickly navigate a web page visually, those users who utilize a screen reader must do so by listening to each option before understanding what actions are available. Consider this when designing your web pages.

Floats

The float property can help build layouts as well and it's quite possible you will run across this property when dealing with styling at some point or another. It can be confusing so spending some time reviewing it could be helpful. Ultimately the property "floats" the content of an element to the left or right and is commonly used to wrap text around images [2].

The clear property specifies what elements can float beside the cleared element and on which side [2]. It has several properties: none, left, right, both, and inherit. You generally want to clear the same direction as the content is being floated. If you float content to the right, then you should clear the content to the right. This would mean that no floating elements are allowed on the right side of the element [2]. Sometimes content is too large for an element so a clearfix must be applied. If there is any overflow the containing element will expand to enclose the large content. The code samples below are examples of two ways to create a clearfix as described by W3 Schools [2]. 

        .clearfix {
            overflow: auto;
        }

        .clearfix::after {
            content: "";
            clear: both;
            display: table;
        }

Understanding the fundamentals of manipulating the DOM is important. It's what modern frameworks and libraries are built upon. The Node class is "an interface from which a number of DOM API object types inherit" [3]. Those being: Document, Element, Attr, CharacterData (which Text, Comment, and CDATASection all inherit from), ProcessingInstruction, DocumentFragment, DocumentType, Notation, Entity, EntityReference [3]. Because these all inherit from the same Node interface they are able to be treated the same way.

Some of the notable properties include:

  • Node.nodeType: returns an unsigned short representing the type of the node (e.g. ELEMENT_NODE: 1, ATTRIBUTE_NODE: 2, TEXT_NODE: 3, ENTITY_NODE: 6, DOCUMENT_NODE: 9, DOCUMENT_TYPE_NODE: 10).
  • Node.nodeName: returns a DOMString containing the name of the Node. Differs depending on the node type. For example HTMLElement will contain the name of the corresponding tag, like 'audio', for an HTMLAudioElement, or Text will return a '#text' string.
  • Node.nodeValue: returns or sets the value of the current node.
  • Node.parentNode: returns a Node that is the parent of the current node. Returns null if no parent is available (e.g. if the parent is not an Element).
  • Node.previousSibling: returns a Node representing the previous node in the tree, null if there isn't a node available.
  • Node.nextSibling: returns a Node representing the next node in the tree, null if there isn't a node available.

Media Controls

Show / Hide

People consume a lot of media from the web and over the years there have been different advances allowing people to do just that. As MDN explains, in the early days of the web video files were too large to be consumed given bandwidth and other technological limitations [3]. As the web grew extensions like Flash provided a way to consume media and Flash quickly became popular. These extensions have had issues with security vulnerabilities among other things. Now with HTML5 and modern browsers we can utilize the video and audio tags to provide media playback natively within the browser. We can also utilize services like YouTube for the hosting and streaming of content.

Audio

Both the video and audio tag support the following attributes [1]:

  • src: the URL for the audio or video content
  • autoplay: a boolean determining if the content should play automatically or not
  • loop: a boolean for replaying the content automatically or not
  • controls: a boolean telling the browser to show media controls for the content
  • preload: how much data should be preloaded before the content can begin (none, metadata only, or an automatic amount determined by the browser)
  • muted: a boolean specifying if the content should be muted or not
  • type: a string indicating the MIME (multipurpose internet mail extension) type for the content
  • crossorigin: an attribute to determine if the Origin header should be sent along with the request to gather the content or not

The audio element supports three audio file types: MP3, WAV, and OGG. You can provide multiple source files for the different file types as well as a fallback message if the user's browser doesn't support the audio tag (for example IE 8 and below). Video

The video tag supports the above attributes as well as a poster attribute. This would be a URL to an image to display while the video loads [1]. Several other attributes include: buffered for adding time ranges for the buffered media and playsinline for playing the media back within the element's playback area (though not including this doesn't mean the content would default to full screen) [4]. Like the audio tag, the video tag also supports fallback content in the form of a paragraph. When providing fallback content you can explain to the user that the tag is not supported by their browser. You could also link them to the content on another site or provide more description of the content.

The ability to provide fallback sources is important because different browsers may only support certain file types (which is true for older browser versions). As the years go by more and more standards are supported but providing fallbacks is always a good practice. Some attributes need to be supplied for different browsers to play the media correctly. For example the MIME type must be provided for audio media to play correctly in Firefox.

There are a number of events that are related to media playback. These include [1][4]:

  • playing: Playback is ready after being paused or stopped due to a lack of data.
  • ended: Playback has stopped because the end of the media has been reached.
  • timeupdate: The currentTime attribute has been updated.
  • play: Playback has started.
  • pause: Playback has paused.
  • volumechange: The volume of the media has changed.

Graphics

Scalable Vector Graphics and the Canvas element are two different ways to add graphics and animation to a website. SVGs are scalable so the resolution doesn't degrade as an image changes its size. SVGs are great for icons that need to scale based on device size and screen resolution. The canvas is strictly programmable using JavaScript and is great for things like games and animations. Each of these tools has pros and cons so use each depending on your need. Ben Sinca lists several key differences between the two. He writes that SVGs are resolution independent, support event handlers, are best suited for apps with large rendering areas, slow rendering can be complex, and that they are not suited for games [3]. As Aidan Ryan describes in his blog post, canvas is more of a "fire and forget" or "immediate" model of rendering whereas SVGs have a "retained mode" which allows us to change and re-render the graphics [1].

var, let, const

Show / Hide

var is the most common way to declare and initialize a variable. var declarations are processed before any code is executed in a process called hoisting. var variables have a scope within their current execution context [1]. This scope is either globally (when declared outside of a function) or within the scope of the enclosing function that declares it [1]. Re-declaring a var won't wipe its value [1]. Undeclared variables (those without the var keyword as one example) are globally scoped. Undeclared variables don't exist until the code that assigns to them is executed [1]. Undeclared variables are able to be deleted while declared variables are not. Using strict mode and always declaring variables is recommended.

var hoisting ensures that all variables are processed before code execution. Because variables are able to be declared at any point in the code you can even declare them after their use and the program will still function. It's because of hoisting that this works. It's as if the variable declarations are hoisted above the executing code. However it's recommended to always declare your variables at the top of their scope. [1] Functions are also hoisted up so that they are available when called. However if they are defined as expressions (i.e. var f => function(){}) they are not hoisted and can cause undefined errors [4].

let is another keyword used to declare and initialize a variable. Like var multiple variables can be declared in one statement separated by commas. "let allows you to declare variables that are limited to block, statement, or expression scope in which it is used" [2]. This is different than var because var defines the variable globally or locally within a function regardless of block scope [2].

const is a keyword that functions very similarly to let. You must initialize the variable at the time of declaration because const represents a variable that cannot be reassigned. It is read-only. However if you assign an object to a const variable the properties of that object can change values. "A constant cannot share its name with a function or a variable in the same scope" and it is affected by the temporal dead zone (as let is as well) [3]. const is useful in functional programming.

The temporal dead zone refers to the time between declaring a variable and initializing it (once the initialization is terminated). let variables are hoisted up like var variables however var variables are inherently initialized as undefined where let variables are not. So accessing them prior to initialization will throw a reference error. Using typoeof on a variable in the temporal dead zone will also throw a reference error. [2]

global and local scope

Show / Hide

If a variable is defined outside of a block (i.e. code encapsulated with curly braces {}) then that variable has a global scope [3]. This is actually only true for web browsers; it's different when using JavaScipt with Node.js for example [2]. It's not recommended to use the global scope. A naming collision is one reason (if a variable name is used more than once in several parts of the code) [3]. Variables redeclared using var are at the same global scope as previous declarations so the variable will be assigned the new value and overwrite the old value, so be careful here.

Local scope refers to variables defined within a particular block of code or function (i.e. within curly braces {}). For function scope, variables defined within the function are only accessible from within that function at the time it's called. Using const or let within a block (inside curly braces) causes these variables to be only accessible from within that block.

Functions don't have access to one another's variables even if one is calling another [3]. However, because of lexical scoping, when defining a function within another one, the inner function can access the outer function's variables and all the way up to the global scope. The more global something is the less it has access to because it can't access inner variables and functions.

Closures are a related topic. In short this is when a function returns another function. This can be helpful when you need to reuse a unit of work (or instance a function that does asynchronous work to look up some data based on changing user input) [3]. Closures can also be used to return private variables and these functions are known to be privileged functions [3]. this and events

Show / Hide
this

'this' represents the JavaScript context object in which the current code is executing [1]. When outside of a function 'this' refers to the global object whether in strict mode or not [1]. There are other differences between 'this' in strict and non-strict mode. The value of 'this' is determined by how a function is called. 'this' can't be set during execution and may be different each time the function is called [1]. bind() has been introduced to set the value of 'this' for a function regardless of how it's called [1].

When not in strict mode, and when a value of 'this' isn't set by a function call, 'this' will default to the global context [1]. When in strict mode 'this' will default to whatever it was set to right before the call [1]. The value of 'this' can be passed from one context to another using the call() or apply() method [1]. When the value passed as 'this' to call or apply is not an object, internally a call to ToObject will convert the input (for example a primitive) to an object.

bind was introduced in ECMAScript 5. This binds the context of 'this' to whatever argument was passed into bind. It is permanently bound regardless of how the function is called later. For arrow functions "this will retain the value of the enclosing lexical context's this" [1]. If you use call, bind, or apply with arrow functions the 'this' argument (the first argument) will be ignored. When using those methods with an arrow function null should be passed in for the first argument [1].

When a function is called as a method of an object 'this' refers to the calling object itself [1]. This is true whether the function was defined inline with the object definition or assigned later separately [1]. This is also true for an object's prototype chain. Objects will inherit functions from their prototype and 'this' will refer to the concrete object [1]. The same can be said when a function is invoked as a getter or a setter--'this' is bound to the object [1].

When 'this' is used in a constructor it is bound to the new object being constructed. The variables assigned with the new statement will have the properties set on 'this' during construction. If an object is returned from the constructor method then that object is bound to the variable assigned with the new statement. In that case any references to 'this' in the constructor are essentially dead code as it has no affects [1].

For DOM events 'this' is set to the element that fired the event. Although some browsers don't follow that convention for listeners added with methods other than addEventListener [1]. When inline event handlers are used 'this' is set to the DOM element on which the listener is placed [1].

Events

Events are fired in response to actions taken by the user or program. As users interact with a web page for example it can be useful to know that they clicked a button. In response to clicking that button a user may be logged in or a modal might display with relevant information. There are many different types of events. For instance keyboard keys being pressed, clicking, form submission, the loading of a page, or when errors occur. Programming around events is important in any program with user interaction and is not limited to the web. The MDN - DOM Event Reference below lists all of the different events that can occur in the browser [3].

Each event has an "event handler" which is "registered" when a block of code is defined to be run in response to an event [2]. Listeners wait for an event to occur and the handler runs the specific code in response. "It's important to note that web events are not part of the core JavaScript language. They are defined as part of the JavaScript APIs built into the browser" [2].

Event handlers can be registered to elements using JavaScript. First one needs a reference to the element and from there one can assign functions to the various event handler properties (i.e. onblur, onkeypress, onmouseover). We are also able to assign event handlers inline within the HTML code (as event handler HTML attributes), however that's not recommended [2]. This is because it can quickly become unmanageable and inefficient [2]. The current recommended mechanism is to use the addEventListener() and removeEventListener() functions to add and remove event handlers. This lets us easily add and remove different event handlers for different purposes. These methods also allow us to apply multiple event handlers to the same event handler property. Trying to assign two different functions to the same property normally overwrites the first because the event property is assigned a new value. When using addEventListener we can get around this limitation.

Event handlers often pass in the event object to the handler method. This is generally seen in the parameter list named as "e". This event object has a reference to the element that just fired the event. You can access this object through the event object's "target" property. [2]

Preventing default behavior is something that you will run across from time to time. One example that MDN describes is preventing a form from being submitted when a user clicks the submit button. You would want to prevent the user from submitting a bad form and instead display validation information before actually submitting the data to the server. Whenever you have custom logic you would like to apply and circumvent the default behavior use the event object's "preventDefault()" method to stop it from occurring. [2] You can also set the event's return value by setting the event.ReturnValue property. Finally returning false for a given event will prevent the form from being submitted (i.e. onsubmit="return false" where false would be the returned value from a method call).

Bubbling and capturing are two advanced event topics that are important to understand. If an element causes two events to fire and each affects one another negatively this will need to be addresses in your code. For example, if an element should hide when an area of the screen is clicked but also play a video in response to a similar event that is also triggered at the same time (as MDN uses in their example [2]). When events are fired there are two phases. The "capturing" of the event phase which runs an event handler if one is registered in response. The capturing phase checks each ancestor of the element triggering the event and works its way from the outer most ancestor (the html element) down to the closest parent until it stops at the original element. In the "bubbling" phase the opposite occurs. It checks to see if the element that triggered the event has an event handler which should respond and moves up the hierarchy (all the way up to the html element). By default event handlers are registered during the bubbling phase [2]. To prevent issues like this the stopPropagation() method can be used. It will stop the bubbling up process so other handlers are not run. This can be invoked by calling the stopPropagation() method on the event object.

The addEventListener method has another parameter which determines if the event should execute during the capturing phase or not. It is default to false and thus the bubbling phase is when the event will fire. MDN notes that if the event listener is "attached to the event target, the event is in the target phase, rather than the capturing and bubbling phases. Events in the target phase will trigger all listeners on an element in the order they were registered, regardless of the useCapture parameter" [4]. The addEventListener method allows you to add multiple listeners to the same element rather than setting a specific property on an element, for example onload (which only allows you to set one method).

Event delegation is another related concept. Instead of registering handlers on child objects you can instead register an event on a parent element. For example if you have a series of list items. If you wanted a message to pop up when an item is clicked you could register an event listener on the parent element instead. The event would then bubble to the items [2]. This can be helpful if child elements are dynamically changing. The event delegation technique allows the parent to handle the event and determine if a child is affected or not, then responds appropriately.

Here's a historical note MDN points out: "why bother with both capturing and bubbling? Well, in the bad old days when browsers were much less cross-compatible than they are now, Netscape only used event capturing, and Internet Explorer used only event bubbling. When the W3C decided to try to standardize the behavior and reach a consensus, they ended up with this system that included both, which is the one modern browsers implemented." [2]

Custom Objects

Show / Hide

A JS object is a collection of properties and a property is an association between a key and a value [1]. These values can be primitives or even whole functions. You can create a new object like so: var myObject = new Object(); and then you can assign properties and values to it using dot-notation. For example, myObject.name = "My new object"; If you declare a property with no value it will default to the value of undefined. Most of the time properties will be accessed using dot-notation however you can also access and set them using bracket notation. For example myObject['name'] will return "My new object" which we defined earlier. Objects are sometimes called associative arrays [1]. You can even iterate over objects like an array to access the different property values (for example MDN uses a for...in loop to print out each property and its value in their example) [1]. There are actually three built-in ways in ECMAScript 5 to enumerate the properties of an object. The for...in loop (includes all enumerable prototype chain properties), the Object.keys(o) method (which does not include prototype properties), and the Object.getOwnPropertyNames(o) method which returns all property names on a given object (enumerable or not) [1].

There are several ways to create new objects. You can use the object initializer or create a constructor method. It's easy to new up an object using the initializer. Remember that objects are just collections of properties. You can define these by hand by declaring key and value pairs. For example: const myObject = { propertyOne: valueOne, propertyTwo: valueTwo }; The values can be any type (for example a number or a string value). These objects are of type Object (as if a call to new Object() were made) [1]. Object initializers are expressions and each time it's called this results in a new object being created and these distinct objects won't compare to each other as being equal [1].

You can also define object types by defining constructor methods. By convention we capitalize constructor methods. We can then use the "new" keyword to create a new instance of the object. To create a constructor simply define a function with a capitalized name which represent the object type. Within this method you can assign properties to the new object and assign values to them either by taking in arguments and assigning them to the properties or by assigning them in a different way (for example by hard coding initial values or by calling other methods which return a value). The keyword "this" is used to assign values to the object's properties. For example, function Person(name) { this.name = name; }; This constructor creates a Person object and takes one argument which is assigned to the property "name".

Constructor methods can be called from the objects themselves because the constructor is a method attached to the object. So you are able to create a new Person object with the following syntax: var anotherPerson = new personOne.constructor('New Name'); Just remember to use the "new" keyword and pass in required parameters. This can be useful if you don't have access to the original constructor for whatever reason [2]. It can also be useful if you want to know the name of the original object that it is an instance of. You can find this using the name property, like anotherPerson.constructor.name. However this value can change for a variety of reasons (for example transpilers, binding, and inheritance) so using the instanceof operator is better in some cases, as explained by MDN [2].

Remember that we can add properties to objects by simply using dot notation. For example person.age = 10; This would add the age property to the existing object named person. However it won't affect other existing Person objects, only this one [1]. To affect all Person objects the Person constructor would need to be updated to include the new property.

Finally we can create objects using the Object.create method. "This method can be very useful, because it allows you to choose the prototype object for the object you want to create, without having to define a constructor function" [1]. Create makes a new object using the given object parameter as the prototype. So the new object's __proto__ property will refer to the passed in object and its available members [2].

As was described earlier object properties can be accessed by either dot notation (myObject.name) or bracket notation (myObject['name']). You are also able to index into object properties using indexed bracket notation (for example myObject[2] = "Hello World!"). Once you define a property by either indexing or using a property name you can only access it using the same technique (so you couldn't access the last property using dot notation as in myObject.2). The exception is array-like objects reflected from HTML [1]. "For example, if the second FORM tag in a document has a NAME attribute of "myForm", you can refer to the form as document.forms[1] or document.forms["myForm"] or document.forms.myForm" [1].

As MDN explains, "prototypes are the mechanism by which JavaScript objects inherit features from one another" [1]. JS is a prototype based language. This means that inheritance is achieved through the use of prototype objects. These are templates that the "child" object can inherit methods and properties from. The "base" prototype objects may inherit functionality from yet another prototype which can link many classes together. This is often referred to as the prototype chain. "The properties and methods are defined on the prototype property on the objects' constructor function, not on the object instances themselves" [1]. By walking up this chain we can access the various properties and methods available to us [1].

All objects inherit properties and methods from the Object prototype by default. So each object has members like watch and valueOf [1]. When a member that's inherited is called the browser first checks to see if the current object has a matching member to call, otherwise it goes up the chain of prototypes until it finds a match [1]. These members (properties and methods) are not copied to the child objects--they are accessed by walking up the prototype chain [1]. The methods and properties that are inherited from the prototype object are only the members on the prototype object, and not ones defined directly on the parent object itself. So for example as MDN explains, Object.keys() and Object.is() are not available to the inheriting class because those are defined on the Object constructor itself. However members such as watch() and valueOf() are available because these are defined on the prototype object.

To modify the prototype chain of an object simply access the prototype property and add a property or method using dot notation. For example: person.prototype.fullName = function() { return this.firstName + ' ' + this.lastName; }. This will add a method called fullName to all Person objects. This is true even if you've already created an object of type Person. Its prototype will be updated and the new method will be available. You can also do this for properties but that is only useful for static fields since these are shared among all instances of an object type. MDN explains that using the constructor method to define properties and then adding methods to the object's prototype separately is a good practice to separate and group the logic [1]. Being able to modify an object's prototype, which affects all instances of the same type, is quite a powerful and useful feature.

Inheritance in JavaScript

Show / Hide

Inheritance in JavaScript is accomplished within constructor methods. We have to utilize the call() method within the constructor. The child object must define a constructor which calls the parent object's call method. The call method takes a parameter which specifies that value of 'this' when the function runs. The rest of the parameters are the parent's regular constructor arguments.

As MDN explains there is more to do to properly set up inheritance. We also need to set the child's prototype to be an object of the parent's type. For example, Warrior.prototype = Object.create(Person.prototype); The create method will return an object which has a prototype of the same type as the parameter passed in. By doing this we'll ensure the child prototype has all the methods available from the parent object's prototype. The last thing we must do is make sure that the child constructor returns an object of its type, and not its parent's type. We accomplish this by adding one more line of code: Object.defineProperty(Warrior.prototype, 'constructor', { value: Warrior, enumerable: false, writable: true }); This will ensure we create an object of the desired type. Finally to define new functionality for an existing method inherited from the parent simply define the same method on the child's prototype. All of what has been discussed is just one way to implement inheritance. Frameworks and new features of ECMAScript allow for classes and other common constructs like the extends keyword. [1]

As MDN summarizes, there are three major groups of object members to remember. Ones defined inside of a constructor method that are given to object instances. Ones that are defined on the constructors themselves which are only available on the constructor. And finally ones that are on a constructor's prototype which are accessible from instances and inheriting objects. [1]

ECMAScript2015 added class syntax which is much like what you would find in languages such as Java, C#, and C++. If you have a background with those OO languages you will quickly understand the class syntax in ECMAScript2015. Classes have a constructor method, class methods, and properties. MDN explains that underneath it all the class syntax is being converted to prototypical inheritance models (class syntax is just "syntactic sugar" to reduce code and feel more consistent with other languages many developers utilize) [1]. To inherit using this syntax we use the extends keyword. For example: class Warrior extends Character { constructor() { super(); } } The call to super is important and must be done first. This way the parent class will be able to initialize 'this' and correctly set up any references [1]. Remember to pass along any required parameters to the parent's constructor by passing them into the call to super(). Adding getters and setters is also common depending on the intended use of your object. These return and set values of instance fields and ECMAScript2015 has syntax for creating them. To add a getter and setter to the Warrior class from above use the following syntax (which also requires an _attackPower field on the class):

Getters and setters are common patterns for setting and returning a class field value. This paradigm has been around for a long time. Not every field needs a getter or a setter and if you are adding these you might want to spend a little more time considering why you are adding them and if another mechanism would be better (as far as encapsulation and abstraction goes). [1]

Inheritance is a powerful tool which allows for reuse. Functionality that we want to be reused among related but subtly different objects can be inherited from a base class and then extended for custom behavior. Too much inheritance can add unnecessary complexity to a project. Depending on your need composition may be a better choice. In the end inheritance is just another tool in your toolbox and shouldn't be the hammer for all of your problems. There are a lot of benefits from using it but it has its place like any other tool. Application Cache

Show / Hide

The application cache can be used to store files offline and reduce server load [1]. However MDN does not recommend using this feature as the application cache is being removed from the web standard and should no longer be supported. Service workers are encouraged instead. That said here is a bit of information on the topic in case you run into them in legacy applications.

When using the app cache feature you must define and specify a manifest file. This file is then set within the HTML tag's attribute list using the manifest attribute. Each page that you intend to cache should include this attribute. You can however specify specific pages within the manifest file itself to cache them (known as master entries) [1]. The browser automatically adds visited pages to the cache if the manifest attribute is set. MDN notes that some browsers notify the user and ask if they want to allow caching when the browser detects a site that uses the cache. In general the browser checks if the cache exists, loads cached files, then checks if the server has an updated manifest file and downloads that and any new files listed. In order to clear the cache manually you need to clear out the browsing history for your particular internet browser.

Manifest File

This file has a MIME type of text/cache-manifest. The files can have comments using the pound/hashtag symbol (#) at the beginning of the comment. The files must have three sections: CACHE, NETWORK, and FALLBACK. If a header is not specified it is defaulted to CACHE. The browser loads a new manifest if the file has changed (byte for byte) so including a version number in a comment can be helpful so it can detect a change. Also MDN notes that you should never include the manifest file itself in the manifest for caching because then it is nearly impossible to update it. [1]

  • CACHE: Files listed here are explicitly cached.
  • NETWORK: These files require a connection to the server. Requests bypass the cache. The wildcard character * can be used once. Useful for security reasons.
  • FALLBACK: Specifies fallback pages the browser can use if a resource is not accessible. These include the initial resource to attempt to load and a fallback uri.

The app cache has a number of states including: UNCACHED (cache object is not fully initialized), IDLE (cache is not currently being updated), CHECKING (fetching and checking for updates), DOWNLOADING (change in manifest has caused new files to be downloaded), UPDATEREADY (new version is available for download), and OBSOLETE (cache group is obsolete). [1]

Flexible Box Model

Show / Hide

We've already seen examples of the flexbox display type and many of its properties above in the Layout Container section. Here are some more details from the W3C documentation.

As the official documentation explains, the flex layout allows content to be laid out to flow in any direction (left, right, up or down), allows for rearranging the element ordering, allows elements to dynamically resize given the allowable space, can be laid out along either a horizontal (main axis) or vertical (cross axis) and can also be aligned in relation to siblings along the cross axis [1].

Section 2 of the official documentation outlines various terms to know when discussing the flexbox layout [1].

  • main size: the width or height of a flex container or flex item, whichever is in the main dimension.
  • cross size: this functions the same as the main size but for the cross (perpendicular) axis.
  • main axis: the primary axis which flex items are laid out on, this extends from the main-start to the main-end.
  • cross axis: the axis which is perpendicular to the main axis. This extends from the cross-start to the cross-end.
  • main/cross-start and end: flex items are placed into the container beginning from the main-start and cross-start sides and extending toward the main-end and cross-end sides.

There are two flex display properties. "flex" which generates a flex container that is block-level and "inline-flex" that generates a flex container that is inline-level [1]. While the flex property behaves similarly to a block display there are differences. For instance the float and clear properties have no effect. Likewise for vertical-align. This is also true for the pseudo-elements ::first-line and ::first-letter.

Anonymous flex items are created from text runs that are not wrapped in an element. White space is removed and the text is placed inside of an unstylable anonymous block container flex item [1].

Grid Alignment

Show / Hide

We discussed this topic within the Layout Container section above but here are some more details.

In the past using HTML tables for layout was common. It provided a way to organize content but it is not very flexible and is cumbersome. Now we have ways to achieve similar results but in a much more modern way which is responsive to a user's screen size. We can use CSS to define that we will use a grid layout and then we can define the number of columns and rows to use within that grid. This is now fully baked into native CSS. At one point Microsoft had specific extensions of their own for use within IE and Edge (i.e. -ms-grid).

A grid can either be block-level or inline (which is set using grid or inline-grid for the display property). Grid items are not affected by some properties like float, display: inline-block, display: table-cell, vertical-align, or column-* [3]. When defining the columns and rows you can name each line of the grid if you'd like. The column and row properties define the size and spacing of the grid. You can even use fractional units using the fr unit [3]. Fractional parts are calculated after any non-flexible items to determine the correct space available.

The grid-area property specifies a name for an item and the grid-template-areas property can reference these grid-area names to define a template and determine if elements span multiple spaces (which is set by repeating the name of an area multiple times in a row) [3]. grid-areas can also define a particular area within the grid which uses four values (grid-row-start, grid-column-start, grid-row-end, and grid-column-end). When defining grid-template-areas you must be sure that the template lists the same number of cells in each declaration. Empty cells can be specified with a period character (i.e. ".").

grid-templates are shorthand for grid-template-rows, grid-template-columns, and grid-template-areas. You can specify the rows and columns and default the areas to none, or even set all three properties with a slightly more complex syntax [3]. However CSS-Tricks.com recommends using the grid property instead because it implicitly resets the grid-auto-columns, grid-auto-rows, and grid-auto-flow properties.

There are also controls for setting the space between the grids. You can do this by setting the grid-column-gap and grid-row-gap properties. There are ways to justify the content using the justify-items property (center, stretch, start, and end) and you can align items with the align-items property (center, stretch, start, and end) [3]. The entire contents of the grid can also be justified or aligned using the justify-content and align-content properties.

The grid-auto-flow properties tell the grid to either fill in more rows, columns, or empty space automatically (using the row, column, or dense values). The grid property is shorthand for several properties including grid-template-rows, grid-template-columns, grid-template-areas, grid-auto-rows, grid-auto-columns, and grid-auto-flow.

Regions, grouping, and nesting layouts

Show / Hide

Infocodify.com explains that regions are "a new CSS3 construct that allows you to have content flow through various regions in a webpage" [1]. In order to use regions the layout must implement two things. Those are a content source and regions which will display that content. This content can come from another web page with the use of an iframe or from another element on the page. Infocodify's article explains that at the time of their article this didn't work in all browsers [1]. Ultimately these mechanisms allow for content to fill in the defined regions similarly to how a magazine or newspaper is laid out where content is placed in grids and columns and the articles flow from page to page and around other things like advertisements. Using regions allows you to add other elements in-between them and to pull your content from one source but have it placed on the page across all of the defined regions. The regions which the content flows over are called a region chain.

You are also able to control where content breaks and moves to a new region with the break-before, break-after, and break-inside properties. So you could use these to be sure a heading of an article isn't the last line of a region and instead gets displayed in a new region with the beginning of its content. For instance this could be achieved by setting the break-after and break-inside properties for heading tags to 'avoid', as WebPlatform describes, to be sure the headings are placed within a single region and don't end up alone [4]. The region-fragment property determines how the last region behaves. If the content were to break in the last region the overflow will not be rendered because there are no other regions for the content to flow to but it would act as though it was flowing into the next region and a break would occur.

Nesting and Grouping

Nesting and grouping help organize and reduce CSS. As HTMLDog.com explains, if the CSS and HTML are structured well you won't need to rely on element IDs or classes to target specific elements. Instead you can give an ID or class to a top level element and target the child elements like h1 or p rather than needing to target the h1 and p tags by specific names. For example .parent h1 { color: red; } and .parent p { color: blue; }. This is what is referred to as nesting, the child tags are nested within their parent. Grouping allows you to target different elements which all need to be styled similarly. For instance h3, .example, p { color: green; } will target header 3s, elements with the .example class, and paragraph elements and set their color to green. This is shorter than targeting and styling each of those separately. [5]

Media Queries

Show / Hide

Media queries are used to alter which CSS styles are applied based on screen size and device constraints. So your web page can be styled one way for desktops but differently for phones for example. It also allows the UI to change dynamically if you were to resize a browser window for instance to either decrease or increase the viewport. You can supply many different media queries to be sure your application works for a variety of devices. These can also be useful for adding specific styles for a particular screen size in order to fix the user experience. You can even load the style sheets based on these queries rather than placing all the different styles into a single CSS file. This way you will load the correct styles from an organized file targeting a designated size with specific constraints. A style sheet will download regardless but will only apply if the media query is true [2].

MDN explains that media queries are useful for conditionally applying styles with @media or @input rules, targeting media for the style, link, and source tag and other HTML elements with a media attribute, and for testing and monitoring media states using Window.matchMedia() and MediaQueryList.addListener() methods [2].

Media types include all, print (printers), screen, and speech (synthesizers). This value defaults to all. Print is intended for paged material. Media types describe the general category of a device [2].

Media features include width, height, aspect-ratio, orientation, resolution, scan, grid, update, color, among many others. These help determine what device the user is viewing the media with. These are used within the query to target specific features and requirements. You can also apply logical operators. These are "and", "not", "only" and "," (comma). "And" combines media features within a query and is also used to join media features with media types. "Not" negates a media query. You must specify a media type when using "not". "Only" applies the style when the entire media query matches the device and can be helpful "for preventing older browser from applying a style" [2]. When using "only" you must also supply a media type. The comma "combines multiple media queries into a single rule" so it behaves like an OR operator [2].

In practice you'll create media queries based on various conditions which will make use of the logical operators. Furthermore you'll be able to group the queries with commas to create different situations where a particular style should be applied. Remember that "not" is applied last to a specific media query (meaning it won't apply to a comma separated list--only to the query it is attached to) [2]. Media queries are also evolving and have had different iterations known as levels which improve upon the syntax and change features. For instance in level 4 you can use less-than and greater-than (or equal to) operators to specify ranges rather than check for a min- and max- value (i.e. @media (5em <= width <= 10em) { ... }). Implement inheritance

Show / Hide

We've already discussed CSS selectors and pseudo-elements and pseudo-classes above. There are a number of ways to select elements. By class or id, by element tag, attributes, pseudo-elements and pseudo-classes, combinators, and you can even group multiple selectors together by separating them by a comma. Then you can define a style to apply to each selected element within the grouping. Styles can also be applied using inheritance.

"Inheritance controls what happens when no value is specified for a property on an element" [4]. Some are inherited, like an em tag within a p tag. The color of the text doesn't change inside the em compared to the rest of the paragraph. However some are not inherited. Like border, so applying a border to a paragraph won't also affect an em tag inside of it (as MDN uses as an example) [4]. Also, some frameworks like Sass and less make the syntax even easier for you and compile their syntax down to CSS for you.

A selector's specificity boils down to "how many elements could this match" [3]. MDN explains how you can score specificity using a numeric table of four columns (thousands, hundreds, tens, and ones). Inline styles on a page score 1000. ID selectors score 100 for each id selector contained in the overall selector. You score 1 in tens for each class, attribute, or pseudo-class in the selector. Finally you score 1 in the one's column for each element and pseudo-element in the overall selector. The universal selector, combinators, and the negation pseudo-class give no points [3].

There are four universal property values for specifying inheritance. These are inherit, initial, unset, and revert. Inherit will set the property value to its parent's value. Initial will set the value to the browser's default stylesheet, otherwise it will default to inherit. Unset resets the property to its natural state of either inherit or initial. Finally revert will set the value to the user's stylesheet. [3]

There is an order of precedence when using selectors. Ids and classes have more specificity than elements alone for instance. Remember the styles defined last also have precedence over styles declared earlier but that directly targeting an element using its id will always take precedence over other styles. The order goes elements and pseudo-elements, classes, attributes, and pseudo-classes, and finally id selectors which have the most specificity [5].

We can make asynchronous calls to the server and await its response. Libraries and frameworks like jQuery support these types of interactions. WebSockets are also a popular option.

HTML5 WebSockets API

Show / Hide

"The WebSocket API is an advanced technology that makes it possible to open a two-way interactive communication session between the user's browser and a server" [1]. The interfaces include WebSocket, CloseEvent, and MessageEvent. The WebSocket is what is used to send and receive messages. The CloseEvent is sent when the WebSocket closes and the MessageEvent is sent by the WebSocket when a message is received. The CloseEvent is sent to the WebSocket's onclose attribute listener. It has three properties: code, reason, and wasClean. code returns an unsigned short value indicating a specific code (such as 1000 for Normal Closure, 1011 for Internal Error, and 1013 for Try Again Later). There are many codes which are unassigned and are reserved for future use. reason returns a DOMString giving the reason the server closed based on the server and sub-protocol. wasClean determines if the connection was closed cleanly or not. [5] MessageEvents are used for many things and in our case for WebSockets it's related to the onmessage property of the WebSocket interface. MessageEvents have data (what was sent), origin (USVString representing the origin), lastEventId, source (MessageEventSource - WindowProxy, MessagePort, ServiceWorker), and ports (which is an array of ports associated with the channel the message is sent through) [6].

WebSockets themselves have several properties including: binaryType, bufferedAmount, extensions, onclose, onerror, onmessage, onopen, protocol, readyStatus, and url. Its methods are only close and send. Its events include: close, error, message, and open. These are available through the related properties as well. The constructor takes a url and any protocols. The url uses the scheme wss:// but in some cases you can use ws://. [1, 2]

WebSockets are great for real-time applications such as chat programs. Binary and textual data can be sent over the socket to the server. [4] WebSockets have several ready states including: CONNECTING - 0, OPEN - 1, CLOSING - 2, and CLOSED - 3. [4, 7]

jQuery AJAX

Show / Hide

From jQuery's Ajax introductory guide they explain that "traditionally webpages required reloading to update their content" [3]. Giving the example of web-based email this would mean that users would need to reload their email inbox to see if they received new mail. Making a web page request all of its content over and over is very inefficient and making the user reload a page over and over is not a great user experience. "By 2003, all the major browsers solved this issue by adopting the XMLHttpRequest (XHR) object, allowing browsers to communicate with the server without requiring a page reload" [3]. Ajax allows us to query the server for additional information but not necessarily the entire web page's data again-- just a subset that we can append to the DOM for display for example. This reduces loading times in some cases and makes the user experience more dynamic and real-time.

Ajax stands for Asynchronous JavaScript and XML and the XMLHttpRequest object is part of that technology [3]. As the jQuery guide explains, this new technology allowed for services like Google Maps to be built (so users could actively navigate the map and have it automatically update itself). jQuery's ajax methods abstract the work for you because different browsers implement the Ajax API differently. And while XML is in the name, generally JSON or plain HTML is sent to and from the server for processing. [3] Finally Ajax is not able to load data from across domains but Cross-Origin Resource Sharing (CORS) has allowed Ajax requests to hit different domains. [3]

The jQuery ajax method allows you to "perform an asynchronous HTTP (Ajax) request" [1]. Other methods in jQuery wrap this function (such as get and load) so you may not need to interact with it specifically. In jQuery 1.5 ajax returns a jQuery XMLHttpRequest which extends the native XMLHttpRequest object. The ajax method takes a URL for submitting a request as well a list of settings. [1] These settings "are a set of key/value pairs that configure the Ajax request" [1]. You can also default these using ajaxSetup(). Some of the options include: accepts (the type of response it will accept), async (which allows you to specify if the call needs to be synchronous), contentType (the type of data being sent to the server which defaults to application/x-www-form-urlencoded; charset=UTF-8), data (the data being sent to the server), dataType (the expect data type returned from the server), error (which specifies the function to call when an error occurs), and success (which specifies the function to call when the request succeeds). [1]

Web Worker Processes

Show / Hide

Web Workers are a way to create threads in the browser. As MDN explains these won't interfere with the UI and these can even perform input and output operations [1]. Workers run code contained within a JS file in a new thread. These are run in a separate global context than the current window so they can't affect the DOM [1]. "Using the window shortcut to get the current global scope (instead of self) within a Worker will return an error" [1]. Some workers can be shared across multiple files whereas others are only accessible from the script which first ran it. These are known as Shared or Dedicated workers. These use the SharedWorkerGlobalScope and the DedicatedWorkerGlobalScope respectively.

Some of the restrictions of Web Workers include not being able to manipulate the DOM and you can't use some of the methods and properties on the window object. MDN notes some of the things that are accessible include WebSockets and data storage mechanisms like IndexedDB and the Data Store API [1]. In order to communicate between a Worker and the main thread the postMessage method is used in either case. The onmessage event handler allows you to gather the Message object for use which contains the data being sent. [1] Finally workers can do network requests with XMLHttpRequest but as MDN notes again the responseXML and channel attributes will always be null. [1] They also have access to the navigator and location (read-only) objects as well as the application cache. [3]

Pass data to a Web Worker

Show / Hide

Data can be passed back and forth from the main thread to Web Workers using the postMessage method as has already been discussed. The onmessage and onerror event handlers will trigger when receiving a message or when an error has occurred. The onerror method implements the ErrorEvent interface and the handler has three key fields: message, filename, and lineno. These are the error message, the file in which the error occurred, and the line number where the error occurred. [1, 2]

When creating a Shared Worker (one which multiple files have access to) you communicate with it using a port number. This is done explicitly in the case of Shared Workers and implicitly in the case of Dedicated Workers. You can set up the port using the start method. If two-way communication is going to be required the worker and the main thread must call it. The onmessage method can also be used to set up the port implicitly. In either case this must be done before communication can proceed. [1] For Shared Workers the onconnect handler is set up to fire when a connection is made. From there the port can be pull out of the event object and the onmessage handler can be set up on that port to listen for incoming messages. [1]

Understanding the limitations of Web Workers

Show / Hide

The postMessage method only accepts a string value and it cannot be a function [2]. However this string value can of course be a serialized JSON object or XML which can contain lots of useful information. Web Workers don't have events for when they are started and stopped and you can only stop them with worker.terminate or self.stop. Finally workers are contained within their own global context and thus have no access to the DOM from the original page. [2] The messages passed from one thread to the other are copied and not the original reference from the thread. This can take a lot of time to copy if the payload is large. HTML5 Rocks mentions that you can use Tranferable Objects to quickly copy data from one context to another [3]. Web Workers also don't have access to window, document, or parent [3].

Regular Expressions

Show / Hide

Regular expressions are useful for pattern matching. These can also get out of hand easily so having good tests and comments describing what they represent can be helpful. Regular expressions (or regex as it's known) is found in many programming languages and can even be applied to input HTML elements. You could use regex to validate email addresses for example. We did this already above in the "HTML5 Input Types and Attributes" section.

MDN's article below describes Regex in JavaScript. They explain that you can construct a regex object in one of two ways. First is by using forward slashes to enclose the pattern (i.e. /ab+c/). The other is by using the RegExp constructor method and passing the pattern as a string value (i.e. new RegExp('ab+c')). [1] The forward slash constructor represents a regex literal and is good to use when the regex is a constant value [1].

You can use patterns to match against string values to determine if the given string conforms to a particular pattern. Sometimes you are looking for simple substrings like strings that contain a particular set of characters. For example /cat/ would match strings like "cat", "catastrophe", and the like but would not match "ca t". Regex has many special characters that can enhance the pattern matching capabilities. You can use them to find strings that contain more than one instance of a particular character or that start and end with a particular pattern. MDN lists all of the characters in their guide [1].
